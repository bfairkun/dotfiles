What is HPC?

High-Performance Computing — shared compute clusters for big jobs

  • UMich cluster: Great Lakes
  • Need VPN if off-campus
  • Access via SSH
  • Jobs scheduled by Slurm
  • Shared storage: Turbo & home dir

📖 UMich Great Lakes User Guide ↗

ARMIS = HIPAA-safe compute cluster for sensitive/patient data. Same Slurm concepts, separate resource request.
Note that some of you may need to apply for an account first.

HPC Architecture

Compute nodes often can't reach the internet — keep that in mind when running scripts that try to download things.

Login Node vs. Compute Node

TaskLogin node ✅Compute node ✅
Navigate files, edit scripts✅
Small test runs (<1 min, 1 CPU)✅
Submit/monitor jobs✅
Running Claude Code / tmux✅
Heavy computation (align reads, ML training)❌ please don't✅
GPU workloads (SpliceAI, AlphaGenome)❌✅ GPU partition
Large memory jobs (>4 GB RAM)❌✅
Login nodes are shared by everyone — a runaway job there hurts all users.

Slurm Basics

Submit a job

sbatch my_script.sh
sinteractive --account=<account> \
  --nodes=1 --ntasks=4 \
  --mem=16G --time=02:00:00

Monitor & cancel

squeue -u $USER       # your jobs
scancel <job_id>      # cancel job
sacct -j <job_id>     # job history

Job script template

#!/bin/bash
#SBATCH --account=pi_hastings
#SBATCH --partition=standard
#SBATCH --nodes=1
#SBATCH --ntasks=8
#SBATCH --mem=32G
#SBATCH --time=04:00:00
#SBATCH --output=logs/%j.out

module load python/3.10
conda activate my_env
python my_analysis.py

Claude Code — Installing

Follow the official quickstart — it covers auth too.

📖 code.claude.com/docs/en/quickstart ↗

Both Claude Code and Codex have desktop apps — great experience for local work, recommended if you're just getting started on your laptop.
On HPC there's no desktop. Most stable setup: run the CLI inside a tmux session on a login node — survives SSH disconnects, easy to re-attach.
Avoid unofficial install methods (e.g. npm global install via community guides) — there have been reports of security concerns with some workarounds.

Skills vs MCPs

Both can give Claude new capabilities:

  • Skills = a markdown file Claude reads on demand (/ucsc-api). No server process, zero config. Great for wrapping an API.
  • MCPs = a running server Claude connects to. Better for live/interactive tools (a running kernel, your inbox).
  • For querying a REST API, a skill is often simpler than an MCP
💡 Claude is good at writing both. Describe a capability you want, ask it whether a skill or MCP fits better — it can often build it for you on the spot.

Cookiecutter Project

pip install cookiecutter
cookiecutter gh:bfairkun/cookiecutter-quarto-smk
Why separate them? Notebooks accumulate hidden state and don't parallelize. Keeping heavy processing in code/ and viz in analysis/ gives you reproducibility + interactivity.
Snakemake describes rules as inputs → outputs, runs steps in parallel, skips what's already done, integrates with Slurm. Not every project needs it — but the folder structure pays off even if you skip the pipeline.

Tips from the Field

Workflow

  • Plan → Execute → Evaluate loop (Shift+Tab)
  • Use voice for detailed specs
  • Don't let it go autonomous too long without checking
  • Version control everything — git is your safety net

Code quality

  • Use Quarto / jupytext over plain .ipynb — friendlier for git & context window
  • Keep CLAUDE.md updated — it's free permanent memory
  • Regular refactor passes to trim Claude-generated sprawl
  • Proficiency requires prior manual coding experience — know when you're on thin ice
